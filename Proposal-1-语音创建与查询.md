# Proposal 1：语音创建与查询

## 1. 动机 / 用户故事

**一句话提炼：** 用户在走路、通勤、做饭、开会间隙等不方便操作屏幕的场景下，能用一句话把临时想到的安排说出来，并随时确认系统是否记对了。

当前用户故事信号来自以下真实场景：

| 故事编号 | 场景 | 一句话描述 |
| --- | --- | --- |
| 故事 1 | 职场用户路上录入客户拜访 | "明天上午十点客户拜访，A 座 1503" → 系统创建并回执地址 |
| 故事 2 | 固定习惯用户建立周期提醒 | "每周五下午五点提醒我提交周报" → 系统创建周期提醒 |
| 故事 3 | 用户事后复查刚录入的内容 | "明天上午那个客户地址再说一遍" → 系统回答 A 座 1503 |

这三条故事覆盖了 核心闭环：**录入 → 确认 → 查询**。创建是入口，查询是信任基础——用户敢用语音的前提是"我说过的，随时能问回来"。

## 2. 目标用户

> 有零散事项和周期提醒需求，并且愿意在走路、开会间隙、做家务、通勤、照顾家人等不方便操作屏幕的场景下，用语音管理个人安排的人。

## 3. 现有做法及其不足

用户当前管理个人日程的典型做法：

| 现有做法 | 不足 |
| --- | --- |
| 打开手机日历 App，逐项填写标题、时间、地点、重复规则 | 需要掏出手机、解锁、找到 App、点击多个输入框；走路或单手时操作成本高，多数人选择"等下再记"然后忘掉 |
| 对 Siri / 小爱等语音助手口述 | 现有助手对中文日程的槽位提取不稳定，容易漏掉地点或备注，且没有可回看的文本回执，说完不放心 |
| 写在备忘录或微信发给自己 | 没有结构化时间字段，无法触发提醒，需要人工回看并手动搬运到日历 |
| 靠脑子记 | 零散事项一多必然遗忘，周期事项容易记混"这周到底有没有周报" |

核心不足不是"缺一个日历 App"，而是**录入和确认的门槛太高**：用户需要的不是一个更花哨的界面，而是一条从"想到"到"记好"的最短路径。

## 4. 本期范围与明确不做

### 4.1 本期做

只做一件事：**用户能用语音创建日程和随手记录，并能在事后用语音确认系统正确记录了它。**

具体能力：

1. **创建单次日程**：支持用户在一条语音中说出标题、日期、时间、地点/备注，系统提取槽位并创建。
2. **创建周期提醒**：支持每天、每周、每月三类常见周期规则；用户在创建时说出周期即可（如"每周五下午五点"）。
3. **创建即时记录**：用户说"帮我记住/记一下"后面的内容，系统原样保存，不要求包含时间。
4. **槽位追问**：创建日程时缺必要信息，系统只追问缺失的槽位（缺时间问时间），不让用户从头重说。
5. **记录查询**：支持按最近创建顺序或关键词查询即时记录（如"取件码多少"、"车停哪了"）。
6. **日程查询**：支持按时间范围查询日程（今天、明天、某一天、某一周、某一月、今年、下一条），支持上下文指代查询（"刚才那个"）和字段查询（"地址是什么"）。
7. **创建时的冲突提示**：新日程与已有日程时间重叠时，提示用户但不阻止创建；用户确认后允许保留冲突安排。

### 4.2 本期明确不做

| 不做 | 原因 |
| --- | --- |
| 冲突自动解决（如推荐可用时间） | 创建时只提示冲突，不替用户做时间重排 |
| 批量创建（一句话创建多条日程） | 会增加槽位消歧复杂度，先验证单条创建 |
| 通过 IM 发起创建 | IM 只做回执，语音是唯一创建入口 |
| 在语音中逐条播报完整日程列表 | 查询一次只播一条或少数几条摘要，完整列表通过 IM 回执查看 |

## 5. 关键决策与依据

### 决策 1：语音创建后给什么反馈

| 备选方案 | 选择 | 理由 |
| --- | --- | --- |
| A. 只语音播报 | | 听过即逝，用户不放心 |
| B. 只 IM 文本 | | 语音场景下用户可能不看屏幕 |
| C. 语音短反馈 + IM 回执 | ✅ | 语音确认"已创建"，IM 保留完整结构化记录供事后核对 |

语音反馈只播标题和时间（≤ 15 字），IM 回执包含完整槽位：标题、日期、时间、地点/备注、周期规则、创建时间。

### 决策 2：槽位缺失时的追问策略

| 备选方案 | 选择 | 理由 |
| --- | --- | --- |
| A. 缺任何槽位都拒绝创建 | | 太严格，用户说"明天开会"就应该能创建 |
| B. 缺槽位时用默认值填充 | | "明天开会"默认几点？用户期望不可控 |
| C. 只追问必要槽位，可选槽位留空 | ✅ | 标题和时间是必要槽位；日期缺省为今天，地点/备注可选 |

**必要槽位**：标题（必须有）、日期（可缺省为今天）、时间（必须有，否则无法触发提醒）。
**可选槽位**：地点/备注、周期规则。
追问策略：缺必要槽位时只问缺失的，不要求用户重复已说出的信息。例如用户说"明天客户拜访"，系统问"几点？"而不是"请重新说一遍标题和时间"。

### 决策 3：周期提醒的表达方式

| 备选方案 | 选择 | 理由 |
| --- | --- | --- |
| A. 完整 RRULE（iCalendar RFC 5545） | | 过于复杂，本期用户不需要"每月最后一个工作日"这类表达 |
| B. 简单周期关键词 | ✅ | 每天/每周/每月覆盖 80% 以上的真实需求 |

支持的周期表达："每天 X 点"、"每周 X 点"、"每月 X 号 X 点"、"每周一/二/…/日 X 点"。不支持的例子："每月最后一个周五"、"隔周周三"。这些留到后续。

### 决策 4：冲突检测的行为

| 备选方案 | 选择 | 理由 |
| --- | --- | --- |
| A. 冲突时拒绝创建 | | 用户可能故意安排重叠（如同时有两个可选会议） |
| B. 冲突时静默创建 | | 用户事后发现冲突会更糟糕 |
| C. 提示冲突，用户确认后允许创建 | ✅ | 提示让用户知情，但保留用户自主决定权 |

冲突判断标准：新日程的开始时间落在已有日程的时间范围内，或已有日程的开始时间落在新日程的范围内，已有的日程没有持续时间时，在新日程的时间范围内找到冲突日程，并提醒用户。提示指出冲突的日程标题和时间，不提供自动解决建议。

### 决策 5：查询的表达范围

| 备选方案 | 选择 | 理由 |
| --- | --- | --- |
| A. 支持自然语言任意查询 | | 自然语言理解边界太宽，本期不可控 |
| B. 只支持结构化时间范围查询 + 上下文指代 | ✅ | 覆盖"今天/明天/某天/某周/某月/今年/下一条/刚创建"等确定性查询 |

支持的查询：

- 时间范围：今天、明天、某一天（如"7 月 25 号"）、某一周（如"这周"）、某一月（如"八月"）、今年、下一条
- 上下文指代："刚才创建的"、"上一个"、"那个客户的"
- 字段查询：针对已知日程的单一字段（"地址是什么"、"几点"）

不支持的查询：模糊关键词搜索、跨主题联想、未明确时间范围的全文检索。

## 6. 基本概念与信息结构

### 6.1 核心概念

本期管理两类信息：有一天会提醒你的，和不需要提醒、但要能随时问回来的。

- **单次日程**：一次性的安排，有标题、有具体时间。到时触发提醒，过后进入终态。比如"明天上午十点客户拜访，A 座 1503"。
- **周期日程**：按每天、每周、每月重复的安排。每个周期实例独立处理，修改某一次不影响后续。比如"每周五下午五点提交周报"。
- **即时记录**：不需要时间的随手记。用户说"帮我记住"后面跟的内容，系统原样保存。查询时按最近创建的顺序或关键词返回。比如"快递取件码 3-2-1103"、"停车位置 A130"、"大门密码 7734"。

日程和记录共用同一套创建和查询入口：用户说"记一下/帮我记住"后面的内容创建为记录，说"提醒我/几点/几号"后面的内容创建为日程。查询时系统根据问题判断用户想找的是日程还是记录。

### 6.3 交互模型

```
用户语音 → [槽位提取] → 缺必要槽位？→ 追问缺失槽位
                            ↓ 不缺
                        创建日程 → 冲突检测 → 有冲突？→ 提示冲突，等用户确认
                            ↓ 无冲突                    ↓ 确认/拒绝
                        语音短反馈 + IM 回执
```

查询路径：

```
用户语音 → [意图识别] → 时间范围查询？→ 列出该范围内所有日程
                      → 上下文指代？  → 从上下文窗口定位日程
                      → 字段查询？    → 定位日程后返回对应字段
```

### 6.4 IM 回执格式

每条创建操作的 IM 回执包含：

```
📅 已创建日程
标题：客户拜访
时间：2026-07-22（周三）10:00
地点：A 座 1503
周期：无
创建于：07-21 18:30
```

周期日程的 IM 回执增加：

```
周期：每周五 17:00
下一次：2026-07-24（周五）17:00
```

## 7. 原型 / Demo

本期产品以语音为主要操作入口，不提供 GUI，因此原型以示例对话形式呈现。

如果是以 APP 做为载体的话，首页展示一个大语音图标的 icon，然后再做一个小 icon收纳任务、记录等信息。

### 7.1 单次日程创建（完整槽位）

```
用户：明天上午十点客户拜访，A 座 1503
秘书：好，明天上午十点客户拜访，A 座 1503，已记下。
```

IM 回执：

```
📅 已创建日程
标题：客户拜访
时间：2026-07-22（周三）10:00
地点：A 座 1503
```

### 7.2 单次日程创建（缺时间，追问）

```
用户：明天客户拜访
秘书：明天几点？
用户：上午十点
秘书：好，明天上午十点客户拜访，已记下。
```

### 7.3 周期提醒创建

```
用户：每周五下午五点提醒我提交周报
秘书：好，每周五下午五点提醒提交周报。下次是本周五下午五点。
```

IM 回执：

```
📅 已创建周期提醒
标题：提交周报
周期：每周五 17:00
下一次：2026-07-24（周五）17:00
```

### 7.4 创建时时间冲突

```
用户：明天下午两点安排产品评审
秘书：明天下午两点到三点已有客户电话。仍然创建吗？
用户：没关系，还是创建
秘书：好，明天下午两点产品评审，和客户电话时间重叠，已记下。
```

### 7.5 时间范围查询

```
用户：明天有什么安排
秘书：明天有两条安排。上午十点客户拜访，A 座 1503；下午两点产品评审。
```

### 7.6 上下文指代查询

```
用户：那个客户的地址是什么
秘书：A 座 1503，是明天上午十点的客户拜访。
```

### 7.7 字段查询

```
用户：周报是几点
秘书：每周五下午五点。
```

### 7.8 查询结果为空

```
用户：后天有什么安排
秘书：后天暂时没有安排。
```

### 7.9 创建即时记录

```
用户：帮我记住快递取件码 3-2-1103
秘书：已记下，快递取件码 3-2-1103。
```

### 7.10 查询即时记录

```
用户：取件码多少
秘书：快递取件码 3-2-1103。
```

## 8. 验收标准

"明天上午十点客户拜访，A 座 1503" → 创建日程，语音确认，IM 回执完整。
"明天客户拜访" → 追问"明天几点？"，补充后创建。
"每周五下午五点提醒我提交周报" → 创建周期提醒，反馈下次时间。
"下午四点和 Alex 开会"（当前 15:00） → 日期默认为今天。
"明天下午两点半安排产品评审"（已有 14:00-15:00 客户电话） → 提示冲突，确认后允许创建。
"今天有什么安排" → 列出今天日程。
"明天有什么安排"（无日程） → 回复"明天暂时没有安排"。
"刚才那个地址是什么" → 回答最近创建日程的地址。
"周报是几点" → 回答"每周五下午五点"。
"明天下午三点" → 追问"要做什么？"。
"帮我记住快递取件码 3-2-1103" → 保存记录，语音确认。
"取件码多少" → 匹配并回答对应记录。
"我车停哪了"（无相关记录） → 回复"没有找到停车位置的记录"。
IM 回执包含日程/记录的完整信息，可事后查看。

## 9. 反证条件

以下情况出现时，应重新评估 设计假设：

1. 超过 30% 的创建需要三轮以上追问才能完成 → 说明槽位提取准确度不足，或必要的交互模式需要调整
2. 用户频繁通过 IM 回执发现创建错误并进入 撤销 → 说明创建准确度不达标，需要优先修复而非继续扩展
3. 周期提醒创建使用率低于 10% → 说明用户对周期功能无需求，可以简化为只支持单次日程
4. 查询功能使用率极低 → 说明用户不依赖语音确认，IM 回执可能已经满足需求
