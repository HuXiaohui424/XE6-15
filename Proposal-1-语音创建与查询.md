# Proposal 1：语音创建与查询

## 1. 动机 / 用户故事

| 故事编号 | 场景 | 一句话描述 |
| --- | --- | --- |
| 故事 1 | 职场用户路上录入客户拜访 | "明天上午十点客户拜访，A 座 1503" -->系统创建并回执地址 |
| 故事 2 | 固定习惯用户建立周期提醒 | "每周五下午五点提醒我提交周报" -->系统创建周期提醒 |
| 故事 3 | 用户事后复查刚录入的内容 | "明天上午那个客户地址再说一遍" -->系统回答 A 座 1503 |
| 故事 4（场景式-GPS） | 用户停车后语音记录车位，事后查询 | "车停在 A130" -->系统记下车位和当前位置；"我车停哪了" -->系统回答 A130 |
| 故事 5（场景式-GPS） | 通勤路上闪过念头，快速记下怕忘的事情 | "回家以后提醒我查一下下周三去北京的机票" -->系统创建日程；晚上到家后问"回家要干嘛来着" -->系统回答"查下周三去北京的机票" |

## 2. 目标用户

有零散事项和周期提醒需求，或者说只要你想记录、想提醒、想方便那么你就是我们的用户。

## 3. 现有做法及其不足

| 现有做法 | 不足 |
| --- | --- |
| 打开手机日历 App，逐项填写标题、时间、地点 | 需要打开 App->新增事项->填写各个输入框；太麻烦了！ |
| 对 Siri / 小爱等语音助手口述 | 不够智能、用户说什么就是什么，比如：明天 10 点开会，然后又说明天 10 点要去会儿医院，它都会设置成功。 |
| 写在备忘录或微信发给自己 | 和写在纸上的唯一区别就是没有区别 |
| 靠脑子记 | 好记性不如烂笔头 |

## 4. 本期范围与明确不做

### 4.1 本期做

只做一件事：**用户能用语音创建日程和随手记录，并能在事后用语音确认系统正确记录了它。**

**具体一点就是：**能够创建单次日程（复杂一点就是支持周期日程）以及记录一件事。

再多一点内容就是，支持提醒。

再再多一点就是，支持输出到各个第三方日历产品。

再再再多一点就是，支持输出到 IM（飞书、企业微信）。

好吧，不能再多了。



**说明：**日程和记录的区别就是日程有时间维度的考量。

### 4.2 本期明确不做

1. 复杂的东西不做（周期日程本身就有点复杂，那么复杂的时间节点的周期日程更加复杂【每月最后一周 周六 好好玩】）
2. 特殊的情况不做（跨时区）

先保证核心功能即可。

## 5. 关键决策与依据

| C. 语音短反馈 + IM 回执 | 这个 | 小孩子才做选择，我全都要 |

### 决策 2：周期提醒的表达方式

| 备选方案 | 选择 | 理由 |

| B. 简单周期提示 | 这个 | 每天/每周/每月（简单直接） |

### 决策 3：冲突检测的行为

| 备选方案 | 选择 | 理由 |
| --- | --- | --- |
| A. 冲突时拒绝创建 | | 拒绝创建，太绝情了。 |
| B. 冲突时静默创建 | | 冲突了你不告述我，你和 Siri 有什么区别。 |
| C. 提示冲突，用户确认后允许创建 | 这个 | 嗯……不错，用户的事用户决定 |

## 6. 基本概念与信息结构

**日程：**你今天在什么时间点/时间段要做什么事（持续时间可选）

**周期日程：**按照编程的思路可以理解为循环，循环的日程。

**记录：** 和日程最大的区别就是没有时间维度；举两个例子，“我车停在 七牛云 B102 了”、“有个快递取件码是A-123321“

**IM**：输出端，可以将日程、记录等事项信息输出到支持的第三方平台（飞书、企业微信之类的）。

## 7. 原型 / Demo

如果是以 APP 做为载体的话，首页展示一个大语音图标的 icon，然后再做一个小 icon收纳日程、记录等信息。

## 8. 验收标准

能够通过语音创建日程/记录、查询日程、修改日程、删除日程。

能够将日程/记录输出到第三方平台（日历、飞书等）
